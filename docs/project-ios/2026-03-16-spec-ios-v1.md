# Specdown for iOS & iPad — Technical Specification v1

> This document defines what gets built, how it's architected, and what "done" looks like for the initial iOS & iPad release.

---

## Summary

A native iOS/iPad app that wraps the existing `markdown-viewer/` web codebase in a `WKWebView` shell, with a Swift native layer handling iOS-specific capabilities: file access, Share Sheet, system appearance, and iPad multitasking. Architecture mirrors the Electron desktop app — same web code, new platform shell.

---

## Framework Decision: Swift + WKWebView

**Chosen:** Option A from the brainstorm — Swift native shell with WKWebView rendering.

**Rationale:**
- Maximum code reuse — the same `markdown-viewer/` files load unchanged in WKWebView
- Mermaid, Marked.js, Highlight.js, and Panzoom all work in iOS WebKit without modification
- Architecture mirrors the Electron pattern already proven in the desktop app
- Small app size (~5–15 MB, no bundled browser engine)
- Full access to iOS APIs (Files app, Share Sheet, system appearance, multitasking)

**Rejected alternatives:**
- SwiftUI Native: Mermaid is the hard part — no mature native Swift renderer exists
- React Native: Adds framework overhead for what is a WebView wrapper; against project philosophy
- PWA: No Share Sheet, no Files app integration, Apple's PWA support is too limited

---

## Target Platforms

- **iOS 16+** — covers ~95% of active devices; unlocks NavigationStack, Stage Manager (iPadOS), modern WKWebView configuration
- **Universal binary** — one app for iPhone and iPad, adaptive layout per device class
- **Deployment target:** App Store (release) + TestFlight (beta)

```
│  └─────────────────────────────────────────────┘    │

```

### IPC Contract (JS ↔ Swift)

**Swift → JS** (via `webView.evaluateJavaScript()`):
| Call | When |
|------|------|
| `loadFileContent(name, content)` | After user picks a file or Share Sheet delivers one |
| `applyTheme('dark'\|'light')` | On launch and when system appearance changes |

**JS → Swift** (via `window.webkit.messageHandlers.specdown.postMessage()`):
| Message | Payload | When |
|---------|---------|------|
| `openFilePicker` | `{}` | User taps "Open File" in the web UI |
| `savePreferences` | `{ theme: string }` | User toggles theme |
| `fileLoaded` | `{ name: string }` | Web confirms file rendered (for recent files list) |

## Touch Interaction

### Mermaid Diagrams
Panzoom supports touch events natively. Validate pinch-to-zoom and two-finger pan work in WKWebView. If gesture conflicts arise with page scroll:
- **Approach:** "Tap to activate" — single tap enters interactive mode for that diagram; tap outside exits. Identical to Google Maps embedded pattern.
- Implement by injecting a CSS class on tap and removing it on outside-tap.
- Add haptic feedback on diagram activation (Swift side, via `UIImpactFeedbackGenerator`).

### Touch Targets
All web UI buttons must meet 44×44pt minimum. The web CSS will need an iOS-specific override:
```css
/* Injected by Swift shell when running on iOS */
.btn { min-height: 44px; min-width: 44px; }
```
Inject via `WKUserScript` at document start.

## State Persistence

| Data | Storage |
|------|---------|
| Theme preference | `UserDefaults` |
| Recent files (paths + display names) | `UserDefaults` (JSON array) |
| File bookmarks (for sandbox access) | `UserDefaults` (serialized `NSData`) |
| Last-opened file | `UserDefaults` |

No `electron-store` equivalent needed — `UserDefaults` is the standard iOS mechanism and handles all of this cleanly.

## Web Code Compatibility

The `markdown-viewer/` code currently detects the Electron environment via `window.electronAPI`. The iOS app needs a parallel detection mechanism.

**Strategy:** Inject `window.iosAPI = true` via `WKUserScript` before page load. The web code can then branch:

```javascript
if (window.electronAPI) {
  // Desktop: use IPC
} else if (window.iosAPI) {
  // iOS: use webkit message handlers
} else {
  // Web: use file picker / drag-drop
}
```

Minimal changes to `app.js` — only the "open file" trigger path needs branching. Rendering is identical across all three.
