# Specdown for iOS & iPad — Technical Specification v2

> Revision focused on the adaptive iPadOS shell now that the base iPhone/iOS app is implemented.

---

## Summary

Specdown remains a **SwiftUI + WKWebView** app that reuses the shared `markdown-viewer/` codebase, but the shell now differentiates between:

- **iPhone / compact layout** — phone-first controls, bottom iOS action bar
- **iPad / regular layout** — native SwiftUI split layout with a sidebar and the shared viewer in the detail pane

This keeps one app binary and one rendering engine while giving iPadOS a structure that matches the platform better.

---

## Architecture Decision

**Keep:** the current hybrid architecture.

**Add:** adaptive shell behavior at the SwiftUI layer plus a layout-mode bridge into the shared web viewer.

### Why this is the right move

- Rendering stays shared across web, desktop, iPhone, and iPad
- The native shell owns platform layout decisions
- iPadOS gets a better information architecture without a separate renderer or a second app
- Future iPad-only polish can land incrementally on top of the same bridge

---

## iPadOS v2 Scope

### Native shell behavior

- Use `NavigationSplitView` for regular-width iPad layouts
- Keep the existing full-screen shell for iPhone / compact layouts
- Expose a native sidebar for:
  - opening files
  - opening bundled samples
  - viewing recent file names

### Shared viewer behavior

- Add an explicit **iOS layout mode** (`phone` vs `pad`)
- Hide the bottom action bar in iPad mode
- Restore the embedded viewer's header action row in iPad mode
- Keep the viewer content, tabs, TOC, raw view, Mermaid rendering, and print logic shared

---

## Native/Web Contract Additions

### Swift → JS

| Call | Purpose |
|---|---|
| `setIOSLayoutMode('phone' \| 'pad')` | Switch shared viewer chrome between iPhone and iPad presentation |

### Swift-side observable state

`WebBridge` now acts as both:

1. the JS bridge for `WKWebView`
2. the observable model for the SwiftUI iPad shell

Published state:

- `currentDocumentName`
- `recentFiles`
- `hasLoadedDocument`

This lets the native shell present useful context without moving rendering logic out of the shared viewer.

---

## iPadOS UX Model

### Sidebar

The sidebar is intentionally light:

- **Open** — native file picker
- **Samples** — bundled docs for smoke testing
- **Recent** — last opened document names

This avoids duplicating the whole viewer UI in Swift while still making iPad feel native.

### Detail pane

The detail pane remains the shared `WKWebView` viewer.

In iPad mode, the embedded viewer keeps its content-header controls because the layout has enough room for them. That means TOC, raw/preview, split view, print, and other shared viewer actions stay where they already work.

---

## What v2 Does Not Do

- No separate iPad renderer
- No bookmark-backed reopen-from-recent flow yet
- No multiwindow document model
- No keyboard shortcut pass specific to iPadOS yet
- No native sidebar-driven TOC mirroring yet

Those are good follow-on improvements, but they are not required to ship a credible iPadOS-first shell.

---

## Definition of Done

v2 is done when:

- iPad regular-width layouts use a split shell
- iPhone behavior still works
- the shared viewer switches correctly between phone and iPad layout modes
- docs reflect the new state of the project
- Jest passes
- the iOS simulator build succeeds
